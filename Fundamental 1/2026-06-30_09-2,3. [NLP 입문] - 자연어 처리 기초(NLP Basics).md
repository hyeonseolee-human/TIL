# 09-02 텍스트 데이터의 정제와 정규화

정규화(normalization) : 표현 방법이 다른 단어들을 통합시켜서 같은 단어로 만들어준다.

정제(cleaning) : 코퍼스 데이터로부터 노이즈 데이터를 제거
- 토큰화 전 : 토큰화 작업에 방해가 되는 부분들을 배제
- 토큰화 이후 : 여전히 남아있는 노이즈 제거
- 완벽한 정제 => 매우 어려움 => 합의점 찾기(이 정도면 됐다)

## 1) 정규화의 방법
  #### 1. 규칙에 기반한 표기가 다른 단어들의 통합
  - USA = US
  - uh-huh = uhhuh
    
  #### 2. 대, 소문자 통합
Automobile, automobile (O)  
=> automobile을 찾는 질의(query)로 Automobile도 찾을 수 있다  
  
대문자와 소문자를 무작정 통합? => X (의미가 달라지기도 함)
  - US(미국) & us(우리)
  - Apple(회사) & apple(사과)
  - Bush(사람이름) & bush(덤불)

머신 러닝 시퀀스 모델을 통해 소문자 변환을 정확하게 진행시킬 수 있음  
=> But 모델 훈련에 사용하는 코퍼스가 커뮤니티, SNS 등 편의를 위해 전부 소문자를  
   사용하는 사람들로부터 나온 데이터라면 모델이 제대로 된 훈련을 하지 못함  
=> 통째로 다 소문자로 바꾸는 게(단순화) 가장 실용적인 해결책일 수 있음
      

## 2) 정제의 방법
#### 1. 등장 빈도가 적은 단어
정상메일 스팸메일 분류기 예시
- 100,000개의 메일 중 5개의 메일에만 등장하는 단어 => 분류에 거의 도움이 되지 않음
#### 2. 길이가 짧은 단어(영어권)

##### < 영어권 >
길이가 짧은 단어(영어권) = 대부분 불용어(불필요한 단어) 
=> 길이가 2~3 이하인 단어 제거
  - 원문 : "I was wondering if anyone out there could enlighten me on this car."
  - 정제 후 : "was wondering anyone out there could enlighten this car."
  
##### < 한국어 >
한국어는 한자어가 많음 = 각 한 글자의 의미가 크다  
=> 길이가 짧은 단어 제거가 비효율적
  - 용(龍) <=> dragon
  - 학교(學校) <=> school
     
#### 3. 불필요한 단어(불용어)의 제거
자주 나오지만 분석에 있어서 큰 도움이 되지 않는 단어  
= 불용어(불필요한 단어)  - 예) I, my, me, over, 조사, 접미사  
=> 미리 사전으로 정의하여 정제 ( NLTK와 같은 패키지, 개발자 직접 정의 )
